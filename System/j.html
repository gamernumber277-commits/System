🔴 Critical — Plain Text Password Storage

Your code has a comment: "DB stores 'ICT' as plain text, we use direct comparison". This means real passwords are stored as readable text. If anyone gains access to your database — through a hack, a backup leak, or shared hosting — every admin password is instantly exposed.

PHP has a built-in fix: password_hash() to store it safely, and password_verify() to check it. Your own code even mentions switching to this — now's the time.

When saving the password to DB (one time setup):
$hashed = password_hash("yourpassword", PASSWORD_DEFAULT); // save $hashed into the database column
When checking login:
if (password_verify($admin_pass, $admin['password'])) { // ✅ safe — never compare plain text passwords }
🟡 Medium — Error Messages Reveal Too Much

You show two different errors: "Admin username not found!" and "Invalid admin password!". These tell an attacker which part they got right. If they see "invalid password," they know the username is valid and can focus solely on cracking the password.

✅ Use one vague message for all login failures:
$_SESSION['error'] = "Invalid credentials."; // always the same
🔴 Medium — Unescaped Output in Error Display

In the login form you do echo $_SESSION['error'] directly. Always wrap echoed variables in htmlspecialchars() — it's a fast, safe habit that prevents unexpected HTML from rendering.

❌ Current

✅ Safe
echo htmlspecialchars($_SESSION['error'], ENT_QUOTES, 'UTF-8');
